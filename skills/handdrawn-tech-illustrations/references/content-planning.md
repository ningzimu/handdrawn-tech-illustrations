# Content Planning

Plan article illustrations by usefulness, not by section count.

## Simple Planning

1. Core claim: one sentence.
2. Reader promise: what the visual makes easier to understand.
3. Image-worthy points: the few ideas that deserve images.
4. Visual role: scene, diagram, comparison, evidence view, flow, layer, cover hook, or summary.
5. Visible text plan: words, labels, and short notes likely to help the image.

Do not put the whole article on the image. The image should carry a mental model, proof layer, or hook.

## Optional Visualization Prompts

Use these when the image idea is still vague. Do not force every image into this sequence.

Useful questions:

1. What does this image help the reader understand?
2. Would it work better as a scene, annotated diagram, comparison, flow, evidence view, or metaphor?
3. What objects, UI fragments, notes, or labels would make the point concrete?
4. What should stay out so the image does not become a crowded summary?

Examples:

- "Scattered inputs need structure" -> loose cards arranged into labeled trays.
- "A shared interface reduces repeated wiring" -> cables converge into one controlled connector rail.
- "Reliability depends on checks and recovery" -> a workbench with checklist, logbook, and reset handle.
- "Complex work improves through iteration" -> paper cards moving through attempt, observation, and adjustment.

## Information Richness

For article body illustrations, aim for useful reading support instead of a sparse decorative doodle.

Each strong body illustration usually contains:

- one main visual focus,
- 3-6 short labels, note tags, or short phrases,
- one relationship or composition pattern such as route, comparison, before/after, layer, loop, handoff, scene, or evidence view,
- enough surrounding detail to explain the idea when the reader pauses on the image.

Keep it lighter than a standalone information poster: no long paragraphs, no rigid page shell, no repeated page layout, no oversized primary copy.

## Shot List Template

```text
01 / placement / ratio / mode / key text or label / point / visual approach / visible text
02 / placement / ratio / mode / key text or label / point / visual approach / visible text
```

Example:

```text
01 / cover / 21:9 / refined / 截图变结构 / 视觉稿需要被拆成可编辑语义 / 小型拆包机把截图拆成层级抽屉 / 截图变结构、语义层、可编辑
02 / paragraph after "为什么难" / 16:9 / refined / 结构线 / 像素不是结构 / 像素池旁的结构线标注图 / 像素、结构、误判
```

## What Deserves An Illustration

Prioritize:

- Core mechanism.
- Before/after state.
- Workflow handoff.
- A confusing boundary.
- A recurring mistake.
- A tradeoff.
- A memorable visual hook.
- A summary mental model.
- A dense-but-readable concept map.
- A small comparison or tradeoff that clarifies a confusing boundary.

Skip:

- Background paragraphs.
- Pure definitions.
- Long lists.
- Claims without visual structure.

## Default Counts

- Short article: 1 cover + 1-2 body illustrations.
- Normal article: 1 cover + 2-4 body illustrations.
- Long article: 1 cover + 4-6 body illustrations.
- Xiaohongshu cover request: 1 3:4 cover unless the user asks for carousel.

Do not over-illustrate. Enough is better than exhaustive.

## Cover Copy Hooks

Concrete cover copy usually works better than vague labels:

- `接管`
- `拆开`
- `筛掉`
- `折叠`
- `打通`
- `接住`
- `少走一步`
- `让流程自己跑`

Avoid vague cover copy:

- `全面解析`
- `深度思考`
- `效率提升方法论`
- `技术趋势观察`

## WeChat Square Text

For 1:1 covers:

1. Keep the core verb and object.
2. Drop clauses and explanations.
3. Prefer 4-10 Chinese characters.
4. Use 2 short lines only when needed.

Examples:

```text
从截图到结构化说明，真正难的是页面语义
=> 截图变结构
```

```text
开源了一个 Skill，让 AI 接管你屏幕边那张便签纸
=> AI 接管便签纸
```

## Screenshots Or Product Evidence

If the visual depends on a real product, UI, code, chart, or hardware object, ask once for source assets unless already supplied.

Use source assets as evidence. Do not redraw or fake them.
